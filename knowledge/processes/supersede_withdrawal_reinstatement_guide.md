# Supersedes, Withdrawals, and Reinstatements Guide

## Overview

This guide covers the administrative procedures for modifying, canceling, or reactivating DOB applications. These procedures address situations where the applicant of record changes, a filing must be canceled, or a discontinued application needs reactivation. Understanding these processes is critical for managing application status changes without losing progress or triggering unnecessary delays.

---

## SUPERSEDES: Changing the Applicant of Record

### What is a Supersede?

A **supersede** is the administrative process by which responsibility for an active DOB application transfers from one licensed professional (the original Applicant of Record) to another licensed professional (the new Applicant of Record). The new AoR assumes all obligations, responsibilities, and authority to modify the application, request inspections, and ultimately close the permit.

### When Supersedes Are Needed

**Typical scenarios requiring a supersede:**

- **Original professional is unavailable:** Architect or engineer who filed the application has left the firm, retired, or is otherwise unreachable
- **Project team change:** Client has decided to switch to a different architectural or engineering firm
- **AoR is MIA:** Original applicant is not responding to DOB or unable to sign required documents (common situation noted in team conversations)
- **Successor firm taking over:** A consultant hired mid-project to replace the original design professional
- **Contractor transition:** In some cases, contractor applicants may need to change

### Applicant of Record (AoR) Requirements

The new AoR must be:
- **Licensed** in New York State as a Registered Architect (RA) or Professional Engineer (PE)
- **In good standing** with the state licensing board
- **Authorized by the owner** to act as applicant on the project
- **Able to certify** that the work complies with approved plans or amended plans

### Build Supersede Steps (General Process)

1. **Obtain authorization:** Get written authorization from project owner naming the new AoR
2. **New AoR signs documents:** The incoming architect or engineer must sign the supersede authorization and any compliance certifications
3. **Submit supersede request:** File supersede form or request through Build system with:
   - Original ALJ/job number
   - Original AoR name and license number
   - New AoR name and license number
   - Owner authorization letter
   - Any amended plans (see Plans Requirement below)
4. **DOB review:** DOB examines the request and approves the transfer
5. **System updated:** Build system updates AoR; new professional gains access and authority
6. **Confirmation:** New AoR receives system notification and can begin managing the application

## WITHDRAWALS: Canceling a Filed Application

### What is a Withdrawal?

A **withdrawal** is the administrative cancellation of a filed DOB application. Once withdrawn, the application is removed from active status and cannot be advanced further. A withdrawn application can be reinstated (see Reinstatements below) or a new application can be filed for the same scope.

### When Withdrawals Are Used

**Typical scenarios:**

- **Project cancelled:** Owner decides not to proceed with work
- **Wrong job type filed:** Application was filed incorrectly (e.g., filed as ALT-2 but should have been ALT-1) and must be refiled under correct type
- **Scope change too large:** Original job scope was too narrow; project needs to be refiled with larger scope
- **Applicant authorization withdrawn:** Owner no longer authorizes the original AoR and wants to void the application rather than supersede
- **SWO (Sworn Statement of Work) correction:** Application filed with errors requiring complete withdrawal and refiling (see SWO Guide)

## Impact of Withdrawal

- **Inspections cannot be scheduled** on withdrawn applications
- **Fees may not be refunded** (depends on DOB policy at time of withdrawal)
- **Applicant of record loses authority** to manage the application

## REINSTATEMENTS: Reactivating Discontinued Applications

### What is a Reinstatement?

A **reinstatement** is the administrative reactivation of an application that has expired, been discontinued by DOB, or was previously on hold. Reinstatement brings the application back to active status and allows work to resume on inspections and approvals.

### When Reinstatements Are Needed

**Typical scenarios:**

- **Application expired:** Job did not receive CO within the typical validity period (typically 1–2 years from filing, [VERIFY: what is the actual timeline?]) and was automatically discontinued by DOB
- **DOB discontinued for inactivity:** No activity on the job for an extended period; DOB closed the file administratively
- **Previously on hold:** Application was placed on administrative hold and must now be reactivated
- **Applicant requesting reactivation:** Original applicant wants to resume work on a project

## AMENDMENTS: Modifying Approved Applications

### What is an Amendment?

An **amendment** is a formal modification to an already-filed or approved application. Amendments address scope changes, plan corrections, new compliance requirements, or other modifications that must be reflected in the DOB record.

### Types of Amendments

**Plan amendments:** Changes to the approved construction documents
**Scope amendments:** Expansion or contraction of the originally approved work scope
**Applicant amendments:** Changes to the AoR or other professional team members
**Compliance amendments:** Addition of new compliance requirements not originally included

## ADMINISTRATIVE HOLDS: Placing and Resolving Holds

### What is an Admin Hold?

An **administrative hold** is a status placed by DOB (often by the assigned examiner) that suspends activity on an application. Holds prevent inspections, approvals, and CO issuance until resolved.

### Common Reasons for Admin Holds

- **Missing documents:** Required compliance documents, signatures, or certifications not yet submitted
- **Fee issues:** Application fees unpaid or billing problems
- **Compliance deficiency:** Application violates code or conflicts with other DOB records
- **Applicant conflict:** Issues with applicant's license status, authority, or qualifications
- **Plan conflicts:** Submitted plans conflict with building code or approved drawings

## Pro Tips and Best Practices

[VERIFY: Are there shortcuts, common pitfalls, or best practices GLE has developed for supersedes, withdrawals, reinstatements, and amendments? Any particular examiners or processes that are smoother than others?]

### General Recommendations

- **Document authorization early:** Before requesting a supersede, secure written owner authorization and have the new AoR ready to sign documents
- **Plan updates:** If plans are changing, have amendments prepared before submitting the supersede to avoid back-and-forth
- **Avoid withdrawals if possible:** Withdrawal loses project history; reinstatement can be uncertain. Supersede is usually preferable if AoR change is only issue
- **Reinstatement complexity:** Reinstatement timelines can be longer than expected. Do not assume it is a quick process
- **Amendment scope review:** Before filing an amendment, confirm with DOB examiner that the scope change will be accepted as amendment (vs. requiring new filing)
- **SWO vigilance:** If pro-certs or sworn statements are involved, verify they are properly certified before withdrawal or amendment
